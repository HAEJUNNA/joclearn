# Codex 프롬프트 템플릿

아래 템플릿을 복사해서 필요한 값만 바꿔 쓰면 됩니다.

## 1) 코드 수정 요청
```text
목표: [무엇을 바꾸고 싶은지 한 줄]

맥락:
- 프로젝트: [예: Spring Boot]
- 관련 파일: [예: src/main/java/...]
- 현재 문제: [에러/불편 사항]

요청:
1. 직접 코드 수정해줘.
2. 변경 이유를 짧게 설명해줘.
3. 수정 후 검증 방법(명령어)도 알려줘.

제약:
- [예: 기존 API 스펙 유지]
- [예: Java 17만 사용]
```

## 2) 버그 재현 및 원인 분석
```text
다음 버그를 재현하고 원인 찾아줘.

증상:
- [구체적 증상]

재현 절차:
1. [step1]
2. [step2]

기대 결과:
- [정상 동작]

실제 결과:
- [오동작]

요청:
- 원인 분석
- 최소 수정으로 해결
- 관련 테스트 추가
```

## 3) 리팩토링 요청
```text
다음 파일(또는 모듈)을 리팩토링해줘: [경로]

목표:
- 가독성 개선
- 중복 제거
- 동작 동일성 유지

요청:
1. 먼저 문제점 요약
2. 바로 코드 수정
3. 변경 포인트를 파일/라인 기준으로 정리
```

## 4) 테스트 코드 작성
```text
다음 코드의 테스트를 작성해줘.

대상:
- 파일: [경로]
- 함수/클래스: [이름]

요구사항:
- 정상 케이스 [개수]
- 예외 케이스 [개수]
- 경계값 포함

환경:
- 테스트 프레임워크: [JUnit5 등]
```

## 5) 코드 리뷰 요청
```text
아래 범위를 코드 리뷰해줘: [파일/브랜치]

리뷰 기준:
- 버그 가능성
- 성능
- 보안
- 테스트 누락

출력 형식:
1. 심각도 순으로 이슈 나열
2. 각 이슈에 파일 경로/라인
3. 수정 제안
```

## 6) 문서화 요청
```text
다음 기능을 개발자 문서로 정리해줘: [기능명]

포함할 내용:
- 목적
- 동작 흐름
- 입력/출력
- 예외 처리
- 예시 요청/응답
```

## 잘 쓰는 팁
- "설명만"이 아니라 `직접 수정해줘`를 명시하면 작업이 빨라집니다.
- 파일 경로, 에러 로그, 재현 절차를 주면 정확도가 크게 올라갑니다.
- 제약(Java 버전, 라이브러리, 코딩 규칙)을 먼저 적으면 재작업이 줄어듭니다.
