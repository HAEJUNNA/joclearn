# Entity, VO, DTO 비교

## 기본 정의
| **구분**    | **Entity**         | **VO (Value Object)**    | **DTO (Data Transfer Object)** |
|-----------|--------------------|--------------------------|--------------------------------|
| **의미**    | 고유한 식별자(ID)를 갖는 객체 | 값 자체로 의미가 있는 객체          | 데이터 전달용 객체                     |
| **주요 역할** | 비즈니스 상태를 표현하고 변경   | 개념적 값 표현 (이메일, 주소, 금액 등) | 계층 간 데이터 전달                    |
| **식별자**   | ✅ 있음               | 🆇 없음                    | 🆇 없음                          |
| **불변성**   | 🆇 변경 가능           | ✅ 불변                     | ▲ 보통 불변                        |
| **계층**    | 도메인                | 도메인                      | 애플리케이션 / 프레젠테이션                |
- 참고: DTO 대신 DC(Data Carrier)라고 부르자는 의견도 있음

<br/>

## 상세 비교
| **구분**            | **Entity**             | **VO**                | **DTO**                          |
|-------------------|------------------------|-----------------------|----------------------------------|
| **목적**            | 도메인 객체 표현              | 값 표현                  | 데이터 전달                           |
| **식별자 존재 여부**     | 있음 (ID)                | 없음                    | 없음                               |
| **동등성 비교 기준**     | ID                     | 모든 속성 값               | 모든 속성 값?                         |
| **상태 변경 가능 여부**   | 가능 (Mutable)           | 불가능 (Immutable)       | 보통 불변                            |
| **비즈니스 로직 포함 여부** | 있음                     | 있음 (값 관련 로직만)         | 없음                               |
| **영속성 연관**        | 있음 (DB 매핑)             | 종속됨 (엔티티 내부)          | 없음                               |
| **계층 위치**         | Domain Layer           | Domain Layer          | Application / Presentation Layer |
| **예시**            | Member, Order, Product | Email, Money, Address | MemberDto, OrderResponse         |

<br/>

## 요약
- Entity: "누가(식별자)"가 중요한 객체
- VO: "무엇(값)"이 중요한 객체
- DTO: "데이터를 옮기는" 객체
